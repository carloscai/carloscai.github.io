Wed 03 September 2014 Software

换行符的编码

换行符的前世今生可以参考"回车与换行的由来"。

换行(newline)，也被称为line ending或line break，是一个加在一行文字最后位置的特殊字符，在其后的字符将出现在下一行。 我们的问题在于不同操作系统上换行符的默认编码不同,这里列出常见的几个桌面操作系统：

  • Unix和类Unix系统（Linux，Mac OS X，FreeBSD等）用\n表示换行符；
  • DOS和Microsoft Windows用\r\n表示换行符。
转义字符 名称 ASCII码
\n LF(Line Feed)，换行 0x0A=10
\r CR(Carriage Return)，回车 0x0D=13

现在的文本编辑器能正确识别和编辑这两种换行符的文本文档，例如Vim和Sublime Text等，Windows的记事本除外。不过由于hashbang后必须以\n这种Unix换行符结束等原因，我建议统一换行符为'\n'。

hashbang

以#!符号开头、放在脚本文件首行的语句被称为hashbang或shebang，作用是自动选择对应的解释器，这仅当文件作为可执行脚本直接运行时有效。 需要执行的Python脚本第一行通常是 #!/usr/bin/env python_version

例如，在Ubuntu下，envPy2.py文件内容如下：

#!/usr/bin/env python2
import platform
print("The python version is {} .".format(platform.python_version()))

envPy3.py文件内容如下：

#!/usr/bin/env python3
import platform
print("The python version is {} .".format(platform.python_version()))

脚本运行及其结果：

$ chmod +x envPy2.py envPy3.py
$ ./envPy2.py
The python version is 2.7.6 .
$ ./envPy3.py
The python version is 3.4.0 .

需要注意的是，在某些平台上（例如Ubuntu）hashbang行必须以\n这种Unix换行符结束，不能以\r\n为行的结束。否则文件作为可执行脚本运行时，会得到这样的结果: No such file or directory，而使用命令python scriptName却又正常运行脚本。

所以我建议在代码中统一使用Unix-style line ending，并做相关设定：

关闭git line ending的自动转换

$ git config --global core.autocrlf false

Sublime Text换行符设定

Sublime Text菜单Preferences->Settings-User，加入以下两行：

" default_line_ending": "unix"，
" show_line_endings": true,

这样Sublime Tex新建文件换行符都是\n，而且在编辑器右下角的状态栏可以看到当前文件是哪种换行符格式。

VIM换行符设定

在文件_vimrc（Windows）或~/.vimrc(Cygwin)中添加一行 set ffs=unix,dos,设置以后VIM新建的文档换行符将会是\n，但不会自动转换dos换行符的文档，转换可参考File format
